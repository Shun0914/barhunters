# 提案資料 叩き台（ドラフト）

**目的**: 経営・人事・現場の論点をつなぐ施策の方向性を示すたたき台。スライド化する前提で章立てしている。

**前提となる議論**: ばーはんたーず（3/29）での合意に近い整理、および「人的資本／部門長向け見える化」の検討を継承する。

### コンテキスト：クエスト上のカレンダーと「つまらなさ」の避け方

西部ガスHDの企業クエスト（Tech0 Step4）では、**2026年5月中の経営会議で企画承認 → 6月から全社運用開始**が前提に置かれている。**最終発表（5/20・東京）は、経営会議への起案の直前**に相当する位置づけである（`docs/research/Tech0_BtoC_Bootcamp_Step4_西部ガス様企業クエスト.md`）。

そのため、**最終で見せるべき「動くもの」**は「提案用のクリッカブルだけ」では物足りない。クエスト本文でも QUEST 3 は **「MVP以上のプロダクト」**、FINAL は **Part 3 で MVP ライブデモ**、評価に **プロダクト完成度・部門長想定のUX** が明示されている。

**つまらなくならない置き方**:

- **早期**（中間レビューまで）: Figma／Make で主導線を触れるのは十分あり。ストーリーと顧客起点の合意を取りにいく。  
- **後半〜最終**: 同じストーリーを **Next.js 等で実際に動くMVP**（推奨スタックあり）に載せ替え、**経営・業務・データの3層設計**と **6,000P/10,000P試算（QUEST2）** を**1本のプレゼン**に接続する。

「ハイブリッドのロードマップ」は**クライアント現場の導入フェーズ**用。「Bootcamp の2ヶ月」はそのロードマップの**0番手に当たる濃密なスプリント**と捉えると、クリッカブル止まりで終わらない動き方と整合する。

## 1. いま整理できている課題（要約）

- **評価と現実**: 役職が上がるほど成果は**部門単位**で見え、部門長は**短期で異動**する。現場は業務負荷が高く、**新しい仕組みを受け入れる余白**が小さい。
- **二つの言語**: 人事側の語彙（ワクワク、行動変容、人的資本）と、経営・投資家向けの語彙（投資対効果、説明責任）が**そのままでは接続しにくい**。
- **先行指標の欠如**: 売上などの遅行指標だけでは「現場で今何を変えるか」が説明しづらい。**日々の現場で観測できる先行指標**が欲しい、というニーズ。
- **挑戦とフォロワー**: 声を上げる人はいても**続く人・広がり**が弱い。インフラ／当たり前品質の文化では**新規挑戦がリスク**として読まれやすい。

---

## 2. パラダイムシフト：「顧客起点」

### 2.1 従来になりがちな起点

- **組織起点**: 部門長に何をしてもらうか、制度に何を足すかから始める。
- **人事起点**: エンゲージメント指標、研修、評価項目の改定から始める。

これらは必要だが、**経営と現場の両方に刺さるストーリー**を作るとき、一点でつまずきやすい。

### 2.2 今回置く起点

**顧客起点**: 「当社のサービス／ガスを使ってくれるお客様にとって、より良い体験・信頼・安全・提案とは何か」から逆算し、**そのために現場が日々何をするか**を定義する。

- 現場の「褒める・認める」「挑戦」「跨ぎの協力」は、**目的そのもの**であると同時に、**顧客体験を守り・伸ばすための行動**として位置づける。
- 経営に説明するときも、「人を動かした」ではなく**「顧客価値に効いた行動が増えた」**という鎖で語る。

### 2.3 このシフトが効く理由（提案の芯）

- **現場の納得**: 「また人事の施策」ではなく**「お客様のための行動が見える化される」**と読める余地ができる。
- **経営の納得**: 人的投資が**顧客・事業のストーリー**に接続しやすくなる。
- **指標設計の一貫性**: ダッシュボードやライトなポイントの**何を数えるか**を、顧客価値の下位要素に紐づけやすい。

---

## 3. 解くべき論点（一文で）

**顧客価値の向上に効く現場行動が、日々増えていることを、現場・部門長・経営が同じ物語で理解できるようにする。**

そのために必要なのは：

1. **顧客価値 ↔ 現場行動**の対応表（仮説でよい）の共有  
2. その行動が**観測・賞賛・可視化**される仕組み（**部門長ダッシュボード**を含む）  
3. **先行指標 → 遅行指標（売上・継続・信頼等）**の時間軸つきの説明枠（2027/2028の切り分け等は会議で出た論点として継承）

---

## 4. 提案の方向性（柱）

### 柱A：顧客価値の分解（仮説ベースでよい）

- 例：安心・迅速対応・提案力・現場の声の蓄積・部門横断の解決、など。**会社と現場で言語化ワーク**を一度挟む。
- ここが固まらないまま「ポイント項目」を決めると、また**制度遊び**に見えやすい。

### 柱B：それを支える「見える行動」

- 会議で出た **「褒める・認める」文化の、日々に近い指標化**を、顧客価値の下にぶら下げる（例：「顧客の緊急対応を跨いで支えた」「改善提案が顧客メリットに結びついた」など、**業種に合わせて具体化**）。

### 柱C：部門長ダッシュボード（ツール）の位置づけ

- **主役は顧客価値と現場行動の鎖**。ダッシュボードはその**運用のハブ**。
- 期待する役割の例：
  - **低摩擦**で「いま拾うべき行動・称賛」を記録する  
  - **配分の責任**が見える（誰に何を認めたか、バジェスト感）  
  - **評価との距離**を運用・文言で明確にする（見える化専用の線引き）  
- 参考調査（Bonusly 等）から継承するのは主に **UI/運用の知見**（ライトな土台、操作＝青、意味分離など）。**顧客ストーリーは自社定義**が必須。

### 柱D：経営向けの「次の言葉」

- ワクワクだけではなく、**顧客指標・事業指標との接続の言い方**をセットで用意する（会議の「ワクワクのその先」）。

### 柱E：導入の現実性

- **パイロット部門**、**課長・リーダー層の巻き込み**、評価項目との**自分ごと化**はオプションではなく設計要素として検討リストに載せる。

## 5. 提案パッケージの形（ハイブリッド）とロードマップ

**なぜハイブリッドか**: 顧客起点の**物語（①）**と**運用（②）**だけでは現場の納得が弱く、**ツール（③）**だけでは「またシステム」になりやすい。短い合意 → 触れるMVP → 本番設計、の**段階に分けると意思決定がしやすい**。

**ロードマップが要る理由**: 経営・投資家説明では**時間軸つきのストーリー**（会議で出た 2027末／2028以降の切り分けなど）が前提になる。現場側も「いつまでに何を試し、何を測るか」が見えないと**参加コスト**が払えない。

### フェーズ案（叩き台・日付は要調整）

| フェーズ | 目的 | 主な成果物・活動 |
|----------|------|------------------|
| **0**（数週間） | 起点と線引きの合意 | **顧客→行動→指標**の1枚図（仮説可）、**運用原則**（見える化と評価の距離）、パイロット候補の条件、**下記「動くもの」を同梱** |
| **1**（例：90日） | 観測と習慣の検証 | **ダッシュボードMVP**＋1〜2部門パイロット、**週次／月次の振り返りの型**、ログの質・量のレビュー |
| **2** | スケールと経営接続 | **経営向け1枚**の指標定義、展開部門の増やし方、制度・評価との接続／非接続の再判断、必要なら本開発・連携 |

各フェーズの**出口基準**（例：フェーズ1終了時に「週あたり記録件数」「関与している管理職数」など）を1行ずつ書いておくと、ロードマップが「絵」ではなく**合意可能な計画**になる。

### 提案の時点で欲しい「動くもの」

ハイブリッドでも、**提案資料＋ロードマップだけ**だと「また議論が続く」印象になりやすい。**同じ会で手元を動かせるもの**があると、顧客起点の話が一気に具体になる。

想定しうる**段階**（上から手軽、下へ行くほど工数増）:

1. **クリッカブルプロト**（Figma / Figma Make 等）… 主要画面遷移・文言・配分のイメージをその場で触れる。バックエンドなしでよい。**中盤までの合意形成に最適**。  
2. **デモデータ入りの縦スライス**… 記録→一覧→残ポイントが**ダミーでも一連で動く**。本番連携はしない。  
3. **MVP以上（クエスト仕様）**… 3層設計（経営・業務・データ）と整合した **実装**、**5/20 のライブデモ**で経営ストーリーと接続する。**最終の期待値はここまで上げる**。

**注意**: 「動く」＝**本番と同等**である必要はない。むしろ**何をまだ決めていないか**（連携、権限、指標の確定）を口頭で添えると、期待値のズレが減る。ただし **5/20 時点では「プロトだけ」だとクエストの評価軸とズレやすい**ので、スケジュール上いつ②→③に切り替えるかを早めに決める。

## 6. スライド化しやすいメッセージ案

1. **課題**: 経営と現場の言語がつながらない／先行指標が弱い。  
2. **転換**: 起点を**顧客価値**に置く。  
3. **骨子**: 顧客価値 → 現場行動 → **日々の観測（称賛・記録）** → 部門・経営のストーリー。  
4. **具体策**: 部門長（＋周辺層）向けダッシュボードと運用ルール（**その場で触れるデモ同梱**）。  
5. **次**: 顧客価値の仮説整理、指標のたたき台、パイロット条件の合意。  
6. **ロードマップ**: フェーズ0→1→2と、各フェーズの出口基準。

---

## 7. 次に詰めるべき論点（アジェンダ用）

- 顧客価値の**一次分解**（誰が誰と1時間で言語化するか）  
- 「褒める・認める」を**どの粒度の行動**まで落とすか（インフラ現実との折り合い）  
- ダッシュボードの**MVPスコープ**（画面・ロール・連携しない前提の範囲）  
- 経営説明用の**1枚**に載せる指標の候補（人的資本開示・内部KPIの両面）  
- 評価制度・360度・パイロットとの**接続／非接続**の線引き  
- **提案時のデモのライン**（クリッカブルのみか、縦スライスまでやるか、誰が当日操作するか）
