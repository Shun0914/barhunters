# テスト運用ワークフロー

E2E／結合テストの **実施タイミング・親チケット・不具合の切り分け** をチームで揃えるためのドキュメントです。  
テストケース本文は、用意でき次第 **`docs/testing/`** に置く想定です（現時点ではフォルダ・ファイルは任意で追加してください）。

---

## 目的

- スプリントやマイルストーン単位で、実装済み機能をテストする
- 発見した不具合を Issue で追跡する
- 「実装が正しい／テストケースが古い」を判別しやすくする

---

## テスト実施前の準備

### 1. テストマスター Issue（親チケット）

スプリント終わりやリリース前など、タイミングを決めて **親 Issue** を 1 件立てます。

**タイトル例**

```text
[E2E] スプリント2 テスト実施マスター
```

**本文に含めるとよいもの**

- 目的・スコープ（どの画面／API まで見るか）
- 参照するテストケースドキュメント（例: `docs/testing/e2e_sprint2.md` ができたらリンク）
- 完了条件（全ケース実行済み、不具合は Issue 化済み、など）
- **マイルストーン**: 実装 Issue と切り離したい場合は付けない運用も可（チームで決める）

### 2. 環境の確認（実装後の目安）

チェックリストの例:

- [ ] バックエンドが起動している（例: `http://127.0.0.1:8000`）
- [ ] フロントエンドが起動している（例: `http://localhost:3000`）
- [ ] DB・シードが手順書どおり（`docs/setup/setup.md` や README）

初期データ投入スクリプトがある場合は、そのリポジトリの README に従います（**barhunters** では今後 `backend/` に追加される想定）。

---

## テスト実施手順

1. **テストケースドキュメント**（存在すれば）を開く
2. ケースごとに実行し、結果を記録する
3. 失敗時は Bugfix Issue を切る / テスト側の誤りならテスト改善を Backlog に回す

### 結果の記号（例）

| 記号 | 意味 |
|------|------|
| ✅ Pass | 期待どおり |
| ❌ Fail | 不具合の可能性 |
| ⚠️ Skip | 前提未満で未実施 |
| 📝 Note | ケース本文の修正が必要 |

---

## 問題発見時の対応

### 不具合と判断した場合

1. Issue を新規作成  
   - タイトル例: `[Bugfix] 因果ストーリーで検索後にカードが空になる`
   - ラベル: `bug` および `area:frontend` / `area:backend` 等（リポジトリの運用に合わせる）
2. 本文に **再現手順・期待動作・実際の動き・環境** を書く
3. 親のテストマスター Issue にコメントで関連番号を追記する

### テストケース側が古い／誤りの場合

1. `[Backlog]` や `[Testing]` など分かるタイトルで Issue を立てる  
2. ドキュメント修正 PR または次スプリントのタスクに回す

---

## テスト完了時

親 Issue に **サマリー** をコメントで残すとよいです。

```markdown
## テスト実施サマリー
- 実施日: YYYY-MM-DD
- Pass / Fail / Skip 件数
- 派生 Bugfix: #xx, #yy
- 備考: …
```

親 Issue のクローズ条件はチームで決めます（「全ケース実行」「Critical のみ修正済み」など）。
