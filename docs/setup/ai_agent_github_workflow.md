# AI／コーディングエージェント向け GitHub 運用手順書

この文書は **Cursor（等）で動く AI エージェントが、リモート運用として安全に読み込める**ことを目的にしています。人間向けの一般的な説明は [github_workflow.md](./github_workflow.md) にあります。ここでは **エージェントがやりがちな取り違え（特にマージ経路の誤り）を防ぐ**ことを最優先にします。

対象リポジトリの正: **GitHub `Shun0914/barhunters`**（ブランチのデフォルトは `main`）。

---

## 1. エージェントが持つツールの「実運用」（重要）

このリポジトリで Cursor などのエージェントが **実際に使う経路は次の二本立て**が基本です。

| レイヤー | 手段 | 役割 |
|---------|------|------|
| ローカル | **`git`** | `fetch` / `checkout` / `merge` / `branch` / `commit`／**リモートの作業ブランチへの `push`** など |
| GitHub のリモート操作 | **`user-github` MCP**（`call_mcp_tool`） | Issue の作成・更新、PR の作成・マージ、メタ確認など **GitHub が提供する状態を正とした操作** |

**`gh` CLI について：** 環境によっては **`gh` が入っておらず**、エージェントがそれを実行できないことが普通です。そのため、この文書での **標準運用は `gh` 前提ではありません**。  
ユーザー（人間）の端末には `gh` があり、[github_workflow.md](./github_workflow.md) と同様のコマンドで操作する運用があるかもしれませんが、それは **人向けまたは「`gh` が利用可能なシェル」向けの補助**と捉えます。

ユーザー指示に「GitHub は `gh` と書いてある」場合でも、エージェント環境で `gh` が **command not found** になることがあるため、そのときは **必ず MCP にフォールバックする**。**`git push origin main` だけで PR マージを代替しない**こと（後述）。

### MCP と git を使うときの鉄則

- MCP サーバー **`user-github` が公開している各ツールのスキーマ**（モデルコンテキスト上の MCP ツール説明ファイル）を **`call_mcp_tool` の前に必ず確認**する。
- **PR を `main` に取り込む**ときは、`create_pull_request` の **`base` を `main` に明示**したうえで `merge_pull_request` を実行する（後述）。
- 「変更を取り込む」＝**通常は PR の公式マージ**である。ローカルで `git merge` だけして **`git push origin main` だけで済ませる**のは、**利用者から明示された例外**がある場合を除き、**トレーサビリティとユーザー意図の両面で有害**となる。

## 2. 標準フロー（Issue あり／なし共通の「正」）

ユーザーが「修正して」「PR」「マージ」など **本流への取り込み** を求めている場合の標準です。

```
1. ローカル: main を最新化 → ブランチ作成（命名: feature/issue-{n}-..., fix/issue-{n}-... 等）
2. 実装・コミット
3. git push で作業ブランチのみリモートへ（main へ直接 push しない）
4. GitHub:
   • create_pull_request（head=作業ブランチ、base=main、本文に Closes #{n} 等）
   • （必要ならレビューや CI 待ち）
   • merge_pull_request（merge_method はリポジトリ慣例に合わせる。本リポジトリはこれまで merge コミット併用例あり）
```

### base ブランチの確認

- 新規 PR の **比較元（base）は原則 `main`**。
- GitHub UI または `get_pull_request` で **`base.ref` が意図と一致しているか確認**すること。過去には `feature/*` が base のままの PR が存在し、この状態で `merge_pull_request` をかけると **想定しないブランチにマージされる**。

## 3. Issue の起票・更新

- **バグ単発の報告」「仕様確認用の親チケット」** は `create_issue` を利用する。
- 本文には **症状・期待動作・再現環境・原因候補（コードパス・行の説明）・受け入れ条件**を書けるだけ書く。スクショはユーザーがワークスペースに置いたパス／URL があれば言及する。
- 起票のみで済ませ、「プッシュしない」など **ユーザー側の確認待ち** があれば、その旨も Issue またはチャットで明記する。
- **`update_issue`** で問題の本文・タイトルを差し替える場合、**ユーザーが承認済みの方針**を反映すること（運用ポリシーと矛盾する単独判断はしない）。

---

## 4. 禁止に近いアンチパターン（エージェント向け）

1. **GitHub MCP（`merge_pull_request` / `create_pull_request`）を経由しない**うちに、`main` にだけローカルマージ済みとして **`git push origin main`** だけで済ませ、ユーザー依頼の「PR をマージ」とみなさないこと。**PR は Open のまま・Issue とも紐付かない**など不整合になりうる。
2. **`merge_pull_request` を使わず**に済ませた結果、ユーザーが **`main` 上の状態と GitHub の PR 状態が食い違う**状態を残すこと。
3. **`base` が `main` でない PR** に対して、ユーザーが「マージ」を求めていると解して機械的に `merge_pull_request` すること。**先に base の是正または PR の作り直し**を検討する。
4. ユーザーのユーザー指示にある **文体（例: 敬体）／コミット運用／秘密情報をコミットしない** などを、この文書だけを理由に無視すること。

---

## 5. ドキュメント・コード変更のプッシュタイミング

- **「ドキュメントを書いて確認してから」「プッシュはしない」** などの明示があれば、**コミットもプッシュもしない**。ファイル作成のみでもよい場合はワークスペースに書いたうえで、ユーザー確認後にユーザーまたはエージェントがコミットする。
- 明示がないときはリポジトリの通常ルール（ユーザールール・チーム運用）に従う。
