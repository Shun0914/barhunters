## 1. 目的

西部ガスグループにおける**人的資本・挑戦活動の可視化**と、**ポイント／申請フロー**を通じた記録・承認の基盤を提供する。経営・部門長・中間管理職・一般社員が、**同じ情報を起点に**活動状況を把握・入力できることを目指す（詳細は企画資料の Lean Canvas・ペインポイント参照）。

## 3. スコープ

### 3.1 含む（MVP 優先／〜最終デモ想定）

- **認証**: ログイン・ログアウト（デモは簡易実装可。企画資料: 社員情報 DB 照合・接続方式 TBD）
- **ダッシュボード**: トップ／一覧的な閲覧、**フィルタ・ソート・表示切替**（企画資料・Figma の因果ストーリー等）
- **ポイント入力**: 数値・項目・**タグ**等の入力（企画資料のユーザーストーリー）
- **UI**: Figma に準拠した主要画面（サイドナビ、ヘッダー、メインコンテンツ）

### 3.2 後続フェーズ（企画資料 (P) マーク相当）

- **申請フロー**: 決裁開始、決裁者への通知、承認／差戻し
- **エクスポート**: 図・PDF 保存・ダウンロード
- **本番連携**: 社内通知基盤、本番社員マスタとの同期方式の確定

### 3.3 明示的に本ドラフトでは未割当

- **部門長向けモバイル MVP**（別 Figma・390×844 想定。リポジトリ外資料）を**同一リリースに含めるか** → **要合意**
- **因果チェーン図**（`dashboard/remixed-9d8265c3.html` 等）をプロダクト内に埋め込むかは **TBD**

## 4. 利用シナリオ（要約）

企画資料より抽出:

1. **コミュニケーション小**: 各自記入 → 承認
2. **コミュニケーション中**: チーム MTG の議題化、部門長の月次参加・FB
3. **コミュニケーション大**: 事前申請、1 階層上と 1on1 で承認

**設計上の論点**: 行動変容（プレッシャー軽減）とコミュニケーション活性化（1on1 負荷）のバランス。部門規模・MTG 形式により運用が変わる → **運用ガイドは別紙推奨**。

## 9. 財務・売上系指標の扱い（ロードマップ考慮）

因果ストーリー（因果チェーン）には **売上高・ROIC・PBR** 等の**財務・市場側の値**が登場する。実装以前に、**「どこまでをプロダクトが参照するか」「数値の正（根拠）はどこから取るか」**はロードマップ上の重要論点である。

| 観点 | 考慮事項（ドラフト） |
|------|----------------------|
| 目的 | 経営説明・施策ストーリーとの**整合**が主目的のときと、**実数監視**が主目的のときで、必要な鮮度・精度・権限が異なる |
| MVP／デモ | **企画資料・IR／有報ベースの確定値**や **JSON／DB に載せたスナップショット**でよい場合がある。**出所・時点**をメタデータとして明示できることが望ましい |
| 本番に向けて | **経理・管理会計・BI・データ基盤**のいずれを正典（SoR）とするか、**更新頻度・責任部門**をステークホルダーと合意する必要がある |
| スコープの線引き | **リアルタイム連携まで必須としない**選択肢もあり得る（例: 四半期ごとの手更新・ファイル連携のみ） |

詳細なデータ観点・フェーズ整理は [04_データ要件.md](./04_データ要件.md) の「財務・市場指標のデータソース（ロードマップ）」を参照。**未確定のまま UI のみ先行すると、後工程で数値定義の組み替えが発生しうる**ため、早期に論点表を用意することが望ましい。
