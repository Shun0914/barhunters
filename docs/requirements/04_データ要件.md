# データ要件（ドラフト）

**概念レベル**の整理。物理スキーマ・マイグレーションは実装フェーズで定義。

---

## 1. 概念モデル（概要）

```
Organization（組織）
  └── User（利用者）
        └── Role（経営／部門長／課長・係長／一般 等）

Indicator / StoryCard（因果ストーリー上の指標カード）
  └── タブ・カテゴリ・表示順・外部リンクメタ

PointApplication（ポイント申請）
  ├── タイトル、ジャンル、ポイント数、活動内容
  ├── 承認者（User 参照、後続）
  └── ステータス（下書き／申請中／承認／却下 ※後続）

ActivityLog / RecognitionRecord（将来的な「拾う」記録との統合候補）
  └── MVP コンセプトとの統合は TBD
```

## 3. 個人情報・機微情報

| 区分 | 例 | 取扱い |
|------|-----|--------|
| 要配慮 | 氏名、所属、申請内容（行動履歴） | アクセス制御、ログ最小化 |
| 評価利用 | コンセプト上「評価に使わない」表明と整理版の「承認根拠」の整合 | **文言・運用・DB 上のフラグ**で要合意（U-02） |

---

## 4. データ保持・削除

| 項目 | 方針（案） |
|------|------------|
| 申請・活動記録 | 業務上必要な期間保持。**退職者の匿名化／削除ポリシー**は人事と合意 |
| ログ | 本番は保持期間を定めローテーション |
| デモ環境 | 本番データのコピー禁止。合成データのみ |

## 6. 整合性・制約（案）

- `points` ≥ 0、上限は設定マスタまたは定数  
- `approver_user_id` は **申請者と同一でない**（後続）  
- 組織階層に応じた **閲覧範囲**（部門長は自部門のみ等）— **要設計**
