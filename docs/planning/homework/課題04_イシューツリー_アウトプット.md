## イシューツリー（テキスト版）

```
人的資本投資の効果を、経営の言葉で証明する
│
├─【経営・説明】経営層に何を「証明」として見せるか
│   ├─ 何をKPIとするか（1P=10万円の根拠・積算ロジック）
│   ├─ ダッシュボード・報告の設計（中計6,000P/10,000Pとの接続）
│   └─ 有報・人的資本開示との整合
│
├─【制度・運用】どんな制度で「効果」を拾うか
│   ├─ ポイント付与の対象（1on1、資格取得、業務改善等のカテゴリ）
│   ├─ 申請・承認フロー（部門長の負荷と現実性）
│   └─ 公平性・形骸化防止（MBO等既存制度との役割分担）
│
├─【組織・受容】現場と部門長が動く条件
│   ├─ 部門長の負荷と「やる理由」の設計
│   ├─ 現場の抵抗要因（給与増と効果の見え方のギャップ）
│   └─ 経営層の「何がわかるの？」への答えの具体化
│
├─【データ・IT】何を記録しどう集計するか
│   ├─ データモデル・ER（申請・承認・ポイント・部門）
│   ├─ 集計・可視化の仕様（Next.js/FastAPI/Azure 等）
│   └─ 権限・認証（部門長・経営層の見え方の分離）
│
└─【先行知見・リスク】うまくいく条件と失敗パターン
    ├─ 投資効果定量化・HCROI・ピアボーナス等の事例
    ├─ インフラ企業特有の課題（組織規模・多拠点・安全・規制)
    └─ 成功要因と失敗パターン（形骸化・不公平感・説明不足）
```

---

## どの観点で分解したか

| 観点 | 意図 |
|------|------|
| **経営・説明** | 「証明」の受け手は経営層。何をKPIとし、どう見せるかを最上位に置いた。 |
| **制度・運用** | 効果を「拾う」仕組み（ポイント・フロー・カテゴリ）を設計しないと証明できない。 |
| **組織・受容** | 制度が動くには部門長・現場の負荷と納得が前提。クエストで言う「部門長の負荷と現場の抵抗」を明示。 |
| **データ・IT** | 証明はデータと可視化に依存。3層のうちデータモデル・API・権限をここに集約。 |
| **先行知見・リスク** | QUEST1の事例研究・成功要因・失敗パターンと接続し、設計の前提を押さえる。 |

## 最も重要だと考える論点（最低3つ）

1. **「1P=10万円」の積算ロジックを経営が納得する形で示せるか**  
   - 中計の6,000P/10,000Pは既に設定されているが、1件あたりの効果をどう定義・集計するかが不明。ここが崩れると経営会議での説明が成立しない。

2. **部門長の負荷を増やさずに、申請・承認と効果の「見える化」を両立できるか**  
   - 現場の抵抗・形骸化の主因は「手間が増えるだけ」という認識。入力負荷を下げつつ、経営層に意味のある指標を届ける設計が成否を分ける。

3. **既存制度（MBO・評価・人事）と役割を分け、公平感と形骸化をどう防ぐか**  
   - 新しいポイント制度が「上乗せの作業」や「恣意的な付与」に見えると浸透しない。MBO等との整合と、評価との切り分けをはっきりさせる必要がある。

## 現時点で情報不足な論点（最低3つ）

1. **西部ガスHDの実際の部門構成・人数・申請フローの現状**  
   - 全部署が対象で部門長が主な利用者とあるが、部門数・承認段階・既存の申請文化が不明。設計の現実性を議論するにはヒアリングが必要。

2. **他社・他ガス会社での投資効果定量化・ポイント制度の成否事例**  
   - クエストでも「他社に解決策はない」とあるが、類似業種・規模での試行錯誤（何が効いたか・何が形骸化したか）の有無は未調査。

3. **経営層が「証明」として最低限欲している情報の具体**  
   - 「その分の結果を出せ」の「結果」が、全社合計Pだけか、部門別・カテゴリ別の内訳か、時系列かなど。壁打ち①（3/24）で確認したい。
