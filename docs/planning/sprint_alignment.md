# スプリントの「単一の正」— FigJam ↔ リポジトリ ↔ GitHub

全体スケジュールを **Phase 単位** と **スプリント単位** の2本立てにし、UX → UI → 開発まで認識齟齬なくつなぐための前提をまとめる。

---

## 共有ボード（チームで編集する場所）

Figma はチームプロジェクト内で編集するメンバーごとに有料シートが必要なため、**1つの共有ボードに集約して作業する**運用にする。

- **共有・作業用ボード**: [tech0_all](https://www.figma.com/board/8iqJEXudloO0H9HNkhBRBK/tech0_all)  
  - チーム全員（しゅんすけ・とむ・くぼしー・きみなり）がここでガント・スプリントタスク・宿題などを編集する。
- product-management 等のチームプロジェクト側は参照・計画の正として残し、実際の編集は tech0_all で行う想定。

---

## 2種類のガント

| 種類 | 用途 | 置き場所 |
|------|------|----------|
| **Phase 単位** | 全体像（QUEST1/2/3、壁打ち、5/20）の把握 | FigJam「全体スケジュール」 |
| **スプリント単位** | 週次で何をするか・開発に継承する単位 | FigJam「スプリント」エリア |

- 講義は水曜 → **スプリントは水曜起点の1週間**。
- Sprint 1 = 3/11〜, Sprint 2 = 3/18〜, … Sprint 10 = 5/13〜5/20（最終発表まで）。
- **壁打ち①（1回目）**: 3/24 19:00–20:00（今長谷氏ヒアリング・方向性確認）。

## スプリントを継承する流れ

- **FigJam**: Phase ガント ＋ スプリントガントで「いつ・何を」を可視化。
- **リポジトリ**: 週次 `homework/YYYY-MM-DD_week.md` や `schedule_overview.md` でスプリント番号・期間を明示。僕（AI）に文脈を渡すときは「Sprint 3 の宿題」のように指定すれば、同じスプリント定義を参照できる。
- **GitHub**: 開発に入ったら、Milestone や Project のイテレーションを **Sprint 1, Sprint 2, …** と同じ名前にする。Issue に `Sprint 3` などのラベルを付ければ、FigJam・ドキュメントと齟齬なく紐づく。

同じ「Sprint 3」を、FigJam のガント・週次 md・GitHub の Milestone で共有する = **単一の正**。

---

## 運用のポイント

1. **スプリント番号・名前を変えない**: Sprint 1〜10 を、計画・宿題・開発で共通の単位として使い回す。
2. **僕に継承する**: 「今は Sprint 4」「Sprint 5 のタスクを GitHub に落としたい」のように伝えてもらえれば、こちらのドキュメントとスプリント定義を前提に整理できる。
3. **GitHub に落とし込むとき**: リポジトリの `docs/figjam/schedule_overview.md` やこのファイルのスプリント一覧を参照し、同じラベル・マイルストーン名で Issue / Project を作ると、UX〜開発まで一貫する。
