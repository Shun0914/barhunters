## 0. ステータスと用語

- **役職（Role）**: `部門長 / 課長(マネージャー) / 係長(リーダー) / 一般職員` の 4 段階
- **直属系統**: 申請者から見た直属の上位役職者の系統（例: 一般職員 → 係長 → 課長 → 部門長）
- **承認ルート**: 申請者の役職に応じて自動算出される直列承認の系列（後述 §2.5）
- **申請者**: ポイントを申請する利用者
- **承認者**: 承認ルート上の決裁者（最大 3 名）
- **申請**: `PointApplication` エンティティ 1 件
- **下書き**: `status = draft` の未提出申請
- **申請中**: `status = submitted` で承認待ちの申請（`current_approval_step` で第何承認者待ちかを管理）

### 1.3 通知ベル＆ポップオーバー（共通シェル）

- **配置**: サイドナビ左下のユーザーアバター横
- **バッジ**: ベルアイコン右上に **未読通知の総数** を数字表示（0 件のときは非表示）
- **クリック挙動**: ベル直上に **ポップオーバー** を開く
- **ポップオーバーの内容**:
  - **最新 4 件** の通知を新しい順で表示
  - 各エントリの表示項目: 通知タイトル / 日時 / 送信者アイコン＋名前
  - 「すべて見る」等のリンクは **設けない**
- **エントリのクリック挙動**:
  - **承認依頼系の通知**（§3.3 #1〜#3）→ **S-04 ポイント承認画面に直接遷移**し、該当申請を **詳細パネルに自動選択**（左カラムの一覧でもハイライト）
  - **それ以外の通知**（承認完了・差戻し等、§3.3 #4〜#5）→ **S-06 通知一覧画面に遷移**し、該当通知を **詳細パネルに自動選択**
- **エントリのホバー**: ホバー時に **背景色を変える**（クリック可能であることを視覚的に示す）
- **既読化**: ポップオーバーから遷移 → 遷移先で対象の詳細が表示されることで **既読**となる（§3.8 既読化の挙動と整合）

### 1.2 申請番号（業務 ID）

- **書式**: 6 桁ゼロ埋めの数字（例: `000001`、`400389`）
- **採番**: 申請が `submitted` になった瞬間に自動採番、`000001` から連番
- **画面表示**: 一覧カード／詳細ヘッダーで「申請番号: 400389」のように表示
- **データモデル**: `PointApplication.application_number: string`（§4.2）
- 内部主キー `id`（UUID）とは別に、ユーザー向け表示用 ID として独立させる

## 2. ポイント・入力（IN-01 〜 IN-09）

### 2.1 IN-01｜ポイント申請フォーム表示（S-01）

- **対象画面**: S-01（Figma `1:133` / `point_shinsei.png`）
- **遷移元**: サイドナビ「ポイント申請 > 申請フォーム」、または S-02 申請状況からの再編集（差戻し後・下書き・取消し後）
- **画面構成**（スクショ準拠）:
  - ページタイトル: 「ポイント申請フォーム」
  - カードヘッダー左: 「ポイント申請フォーム」
  - カードヘッダー右: **「下書き再開」** ボタン（**最新日付の下書きを読み込む**。詳細は §2.6）
  - フォーム項目（上から）: タイトル → 活動ジャンル → ポイント数 → 活動内容 → 第1承認者 / 第2承認者 / 第3承認者（3 列横並び）
  - フッター左: **「+ 一時保存」** ボタン
  - フッター右: **「✈ 申請」** ボタン
- **初期状態**:
  - 新規: 全項目空、`status = draft`
  - 下書きから再開: **最新日付（`updated_at` 降順 1 件目）の下書き**を再表示（§2.6）
  - 差戻し後の再編集: 既存値を再表示（差戻しコメントは仕様上保持しない。§3.4 参照）
  - 取消し後の再編集: 取消しにより `draft` に戻った申請を再表示
- **受け入れ条件**:
  - 未認証なら AUTH-01（ログイン画面）にリダイレクト。**MVP では他チーム提供のダミーセッション機構を前提** とし、本仕様書では認証ロジック・ログイン UI は扱わない（§5.0 / Q-50）
  - 第1承認者は **承認ルート §2.5 に基づき自動入力**
  - 申請者の役職に応じて第2・第3承認者は自動入力 or 空欄表示

### 2.2 入力項目（IN-02 〜 IN-07）

| ID | 項目 | UI | 必須 | バリデーション | 備考 |
|----|------|----|------|----------------|------|
| IN-02 | タイトル | 1 行テキスト | 必須 | **最大 50 文字** | プレースホルダ無し |
| IN-03 | 活動ジャンル | ドロップダウン | 必須 | 選択肢マスタ（`ActivityGenre`） | プレースホルダ「ジャンルを選択」。**選択するとポイント数が自動入力される（§2.3）** |
| IN-04 | ポイント数 | 数値入力（サフィックス「P」） | 必須 | 整数、ジャンルマスタの値 | **編集不可（読み取り専用）**。値はジャンル選択により自動セット |
| IN-05 | 活動内容 | 複数行テキスト | 必須 | **最大 500 文字** | |
| IN-06 | 第1承認者 / 第2承認者 / 第3承認者 | ドロップダウン × 3 | 申請者の役職により段数が変動（§2.5） | 申請者と同一不可、上位役職順 | **自動入力（直属系統で一意）かつ編集可**。プレースホルダ: 「承認者1」「承認者2」「承認者を選択」 |
| IN-07 | タグ | — | — | — | **本画面には存在しない**（要件 IN-07 はスコープ外で確定） |

#### バリデーション運用ルール

- クライアント側で即時バリデーションを行い、エラーは項目下に表示する
- サーバー側でも同等のバリデーションを実施し、HTTP 400 でフィールド単位のエラーキーを返す
- 文字数上限・ジャンル別ポイント数は **マスタ／設定で外出し**

### 2.3 ポイント数の自動入力（IN-04）

- **挙動**: 活動ジャンル選択 → 該当ジャンルの `default_points` をポイント数欄にセット
- **編集可否**: **編集不可**（読み取り専用表示）
- **ジャンル変更時**: 新しいジャンルの `default_points` で上書き
- **ジャンル未選択時**: ポイント数欄は空（プレースホルダ「P」のみ表示）
- **データモデル**: `ActivityGenre.default_points: int`（§4 参照）

### 2.4 承認者の自動入力と編集（IN-06）

- **自動入力**: フォーム表示時に承認ルート（§2.5）から第1〜第3承認者を自動セット
  - **第1承認者** = 申請者の **1 階層上**
  - **第2承認者** = 申請者の **2 階層上**
  - **第3承認者** = 申請者の **3 階層上**
- **該当者がいない場合**: その段は **空欄かつ disabled**（操作不可）
- **編集可否**: **編集可**（時期による異動・代理対応のため、申請者が選択し直せる）
- **選択肢**: 各ドロップダウンは **申請者と同一組織（グループ）に所属する当該役職のユーザー** を候補として表示
  - 第1承認者 = 同一組織の係長一覧
  - 第2承認者 = 同一組織の課長一覧
  - 第3承認者 = 同一組織の部門長一覧
  - **MVP の組織モデル（§4.1）では各役職は 1 グループあたり 1 名のみ存在** するため、各ドロップダウンの候補は **実質 1 名**。UI としてはドロップダウンを維持するが、選択肢は 1 件のみ表示
- **入力方式**: 候補リストからの選択のみ。**フリーテキスト入力での確定は不可**
  - キーワードを入力するとリストが部分一致で絞り込まれる（タイプアヘッド）
  - 絞り込まれた候補から選択して初めて確定。リストに無い文字列のままでは確定できない
- **バリデーション**:
  - 申請者と同一ユーザーは選択不可
  - 各承認者間で重複不可
  - 役職と段の対応が崩れない（第1=係長、第2=課長、第3=部門長 — ただし部門長申請時の扱いは Q-01）

### 2.5 承認ルートの自動決定

申請者の役職（=組織階層上の位置）に応じて承認段数と承認者役職が決まる。**最終決裁者は常に部門長**（comment.md より）。

| 申請者の役職 | 第1承認者（1 階層上） | 第2承認者（2 階層上） | 第3承認者（3 階層上） | 段数 |
|--------------|----------------------|----------------------|----------------------|------|
| 一般職員 | 係長 | 課長 | 部門長 | 3 |
| 係長 | 課長 | 部門長 | （該当者なし → 空欄） | 2 |
| 課長 | 部門長 | （該当者なし → 空欄） | （該当者なし → 空欄） | 1 |
| 部門長 | **申請不可**（2026-05-05 確定 / 旧 Q-01） | — | — | 0 |

- **直属系統で一意**に算出する（申請者の所属組織から N 階層上を辿る）
- 各段の対象役職者が **組織内に存在しない場合は空欄かつ disabled**（comment.md「該当者がいなければ空白」）
- 自動入力された候補は申請者が **編集可能**（時期による異動・代理対応に備える）
- **差戻し後の再申請時は承認ルートを再計算しない**。最初の申請時に確定した第1〜第3承認者をそのまま使う（人事異動が起きていても元のメンバーで再承認）
- **部門長は申請不可**（旧 Q-01、2026-05-05 確定）。フォーム表示時に承認ルート段数 0 を返し、UI 側で「現在、部門長の方の申請は受け付けていません」を表示し送信ボタンを disabled にする。サーバ側 `/submit` も HTTP 400 を返す

### 2.6 IN-08｜下書き保存（一時保存）

- **挙動**: 「+ 一時保存」ボタン押下で `status = draft` のまま永続化
- **保有数**: ユーザーは **複数の下書きを保有可能**（comment.md より「最新日付の」表現）。1 件制限は設けない
- **必須項目の扱い**: 下書きでは **必須バリデーションをスキップ**（タイトルのみ最低限必須など、最終ルールは **TBD**）
- **再開動線**:
  - S-01 カード右上の **「下書き再開」** ボタン → **MVP では最新日付（`updated_at` 降順 1 件目）の下書きのみを読み込む**（複数下書きから選んで再開する UI は MVP 範囲外）
  - S-02 申請状況一覧から下書きの行を選択 → S-01 へ遷移して再編集（複数下書きの個別選択はこちらで対応可能）
- **永続化先**: DB（クライアント側ローカルストレージ併用は **TBD**）

### 2.7 IN-09｜申請送信

- **前提**: 全必須項目のバリデーション通過、承認ルートが確定していること（§2.5）
- **挙動**:
  1. クライアント側バリデーション通過
  2. 確認ダイアログ（要否は **要合意**）→ サーバーへ送信
  3. サーバー側で `status = submitted`、`current_approval_step = 1`、`submitted_at` を記録
  4. **`application_number` を採番**（連番、6 桁ゼロ埋め）
  5. 第1承認者へ通知（WF-02）
  6. S-02 申請状況へ遷移、または完了トースト表示（**要合意**）
- **二重送信防止**: 送信中はボタン非活性化、サーバー側で冪等性キー（**要合意**）

## 3. 申請状況・通知・承認

### 3.1 ステータスと状態遷移

```
[draft]
   │ 申請送信(IN-09) ※ application_number 採番
   ▼
[submitted, step=1] ──第1承認者 approve──▶ [submitted, step=2]※段数2以上
   │      │                                   │
   │      │ 取消し(withdraw)                   │ 第2承認者 approve
   │      ▼                                   ▼
   │   [draft]                              [submitted, step=3]※段数3
   │   （application_number は引き継ぐ。     │
   │    再編集して再申請すると同一番号で     │ 第3承認者 approve
   │    submitted に戻る）                    ▼
   │                                       [approved]
   │
   └── いずれかの段で「差戻し(return)」 ──▶ [returned] ※差戻しコメントは保持しない
       （差戻し先は常に申請者まで一気に戻る。
        承認者と申請者の間のコミュニケーションは
        本アプリ外（直接対話・チャット等）で行う）
```

| ステータス | 説明 | 表示画面 |
|------------|------|----------|
| `draft` | 下書き（新規 or 取消し戻し） | S-01（再編集）／S-02 |
| `submitted` | 申請中（`current_approval_step` で第何段かを管理） | S-02（申請者）／S-04（該当段の承認者） |
| `approved` | 全段承認済 | S-02 完了タブ／S-04 完了タブ |
| `returned` | 差戻し（申請者まで戻る） | S-02（申請者）／S-04 履歴 |

> ※ `withdrawn` ステータスは **設けない**。取消し操作は `submitted, step=1` の申請を `draft` に戻す動作で表現する（§3.6 参照）。

### 3.2 WF-01｜申請の送信

- IN-09 と同義。サーバー側で `current_approval_step = 1` の承認待ちキューに登録、`application_number` を採番
- 監査ログ: `who / when / payload` を記録（**要合意**、[04 §8](../docs/requirements/04_データ要件.md)）

### 3.3 WF-02｜通知

- **チャネル**: アプリ内通知のみ（共通シェルのベル＋ポップオーバー §1.3 ／ S-06 通知一覧 §3.8）。**メール / Teams 等の外部連携は実装しない**

#### 通知タイプと CTA の対応

| # | 発火イベント | 受信者 | 通知タイトル例 | 通知 type | CTA 遷移先 |
|---|--------------|--------|----------------|-----------|------------|
| 1 | 申請送信（IN-09） | 第1承認者 | 「承認依頼: ◯◯◯」 | `approval_request` | S-04 ポイント承認画面（該当申請を自動選択） |
| 2 | 第1承認者が承認 | 第2承認者 | 「承認依頼: ◯◯◯」 | `approval_request` | S-04（該当申請を自動選択） |
| 3 | 第2承認者が承認（段数3） | 第3承認者 | 「承認依頼: ◯◯◯」 | `approval_request` | S-04（該当申請を自動選択） |
| 4 | 最終段の承認者が承認 | 申請者 | 「申請承認のお知らせ」 | `approved` | S-02 申請状況（該当申請を自動選択） |
| 5 | 任意の段で差戻し | 申請者 | 「申請が差戻されました」 | `returned` | S-02（該当申請を自動選択） |

> ※ **取消し（withdraw）操作では通知を発火しない**（第1承認前のみ取消し可能で承認者はまだ着手していないため）。`withdrawn` type は廃止。

- **通知データ（共通フィールド）**: タイトル、本文、対象申請 ID、送信者ユーザー ID、通知 type、既読フラグ
- **既読管理**: S-06 タブ「すべて / 未読」、ポップオーバーは未読・既読を区別せず最新 4 件を表示
- **既読化のタイミング**: 詳細パネルに表示された時点で既読（§3.8）

### 3.4 WF-03｜承認・差戻し

- **対象画面**: S-04（詳細仕様は §3.7）
- **画面要素**:
  - 申請内容の参照（IN-02〜IN-06 の表示）
  - **承認ルートの可視化**（人物アイコン＋矢印で第1→第2→第3を図示し、現在の `current_approval_step` を色で強調）— S-02 と同じビジュアル仕様
  - 詳細パネル右上に「↶ 差戻し」、フッターに「↩ 承認」
- **差戻しコメント**: **本アプリでは保持しない**。差戻しに伴う理由共有・対話は **本アプリ外**（直接対話・社内チャット等）で行う運用とする
- **確認ダイアログ**: **不要**（承認・差戻しともワンクリックで実行）
- **権限チェック**: ログインユーザーが当該申請の **`current_approval_step` に対応する承認者と一致** する場合のみ操作可
  - 例: `current_approval_step = 2` の場合、`approver_2_user_id` のユーザーのみ操作可能
  - 不一致なら 403 を返す
- **状態更新**:
  - **承認**: `current_approval_step` を進める。最終段なら `status = approved`、`decided_at` 記録
  - **差戻し**: `status = returned`、申請者まで戻す。`returned_at` `returned_by` を記録（コメントは記録しない）
- **後続アクション**:
  - WF-02 に従い通知発火
  - **承認待ち一覧から当該申請が消え、次の申請を自動選択**（§3.7 参照）

### 3.5 WF-04｜承認履歴参照

- **タブ構成**: 申請者側 `未完了 / 完了 / すべて`（§3.6）、承認者側 `承認待ち / 完了 / すべて`（§3.7）
- **表示項目**: 申請番号、タイトル、申請日時（一覧）／ 詳細パネルの全フィールドと承認ルート可視化
- **詳細**: 承認ルートの履歴は `ApprovalHistory` から取得、現在ステップを強調表示

### 4.1 User / Organization（補足・MVP モデル）

#### MVP 組織モデル（合意済 / 2026-05-04）

- 1 グループの構成: **1 部門長 + 1 課長 + 1 係長 + 8 一般職員**（合計 11 名想定）

- ユーザー・組織のシードデータは **他チーム提供のダミーデータを利用**（本仕様書では用意しない）

## 5. API 概要（仮）

実装フェーズで OpenAPI 化する想定。

### 5.0 前提と規約（環境・スタック・API 規約）

> 詳細なセットアップ手順は [docs/setup/setup.md](../docs/setup/setup.md) を正とする。本節は **仕様面で前提となる環境・規約のみ** を要約する（実装手順の重複は避ける）。

#### 技術スタック

| レイヤ | 採用 | バージョン要件 | 配置 |
|--------|------|----------------|------|
| フロントエンド | Next.js（App Router） | Node.js **20 以上推奨** | `frontend/` |
| バックエンド | FastAPI | Python **3.10 以上** | `backend/` |
| DB（ローカル） | SQLite（`backend/local.db`） | — | Alembic でマイグレーション管理 |
| DB（本番） | TBD（MySQL / PostgreSQL 候補） | — | [05_アーキテクチャ.md](../docs/requirements/05_アーキテクチャ.md) §3 |
| マイグレーション | Alembic | — | `alembic upgrade head` で適用 |

> ※ CLAUDE.md ではバックエンド実行に Anaconda 仮想環境 `barhunters` を使用するルールがある。`venv` を使う [setup.md](../docs/setup/setup.md) との運用整合は別途確認のこと（**要合意**: Q-40）。

#### API ベース URL とパス規約

- すべてのエンドポイントは **`/api/...` プレフィックス** で提供する（§5 のエンドポイント表参照）
- フロントエンドは環境変数 **`NEXT_PUBLIC_API_URL`**（**末尾スラッシュなし**）でベース URL を解決する
- ローカル開発の既定値: バックエンド `http://127.0.0.1:8000`、フロント `http://localhost:3000`
- 本番のオリジン構成は **TBD**（[05_アーキテクチャ.md](../docs/requirements/05_アーキテクチャ.md) §3）

#### CORS

- バックエンドの環境変数 **`ALLOW_ORIGINS`** で許可フロントオリジンを制御
- ローカル開発時は `http://localhost:3000` を含めること

#### 環境変数（仕様に影響するもの）

| 場所 | 変数名 | 用途 |
|------|--------|------|
| `backend/.env` | `ALLOW_ORIGINS` | CORS 許可オリジン |
| `backend/.env` | `DATABASE_URL` | DB 接続文字列（既定は SQLite ファイル） |
| `frontend/.env.local` | `NEXT_PUBLIC_API_URL` | API ベース URL（末尾スラッシュなし） |

#### 動作確認用の運用エンドポイント（仕様外・参考）

- `GET /health`、`GET /api/hello`、`GET /api/db/meta` は **疎通確認用** のため本仕様書のエンドポイント表（§5 後半）には含めない。詳細は [setup.md](../docs/setup/setup.md) §3.1 を参照。

#### 認証（MVP の前提）

- **MVP では他チーム提供のダミーセッション機構を前提** とし、本仕様書は認証ロジック・ログイン UI・トークン発行を扱わない（[Q-50](#7-未確定事項要合意このドキュメントで追跡)）
- すべての API は「現在ユーザー（セッション解決済みの `User`）」を取得できる前提で記述する
- 申請送信時の `applicant_user_id` は **サーバ側でセッションから自動セット**（クライアントから明示送信しない）
- 認可ルール:
  - 申請者は自分の申請のみ参照・編集可
  - 承認者は `current_approval_step` に対応する `approver_N_user_id` と一致した場合のみ approve / return 可（不一致は HTTP 403）

#### タイムゾーン

- **JST 固定**。サーバ・クライアント間の時刻は UTC で授受してもよいが、画面表示は常に JST に変換する
- 日時表示フォーマット: **`YYYY/MM/DD HH:mm`**（例: `2026/05/01 10:25`）

#### ページネーション

- 一覧 API（S-02 / S-04 / S-06）の既定値: **1 ページあたり 20 件、オフセット方式**
- カーソル方式への切替は将来検討（MVP では不要）

#### 監査ログ（スコープ外）

- 監査ログテーブルおよび記録機構は **他チームが別途提供** する前提とし、本仕様書では設計しない（[Q-10](#7-未確定事項要合意このドキュメントで追跡) は他チーム側で追跡）

### 5.1 エンドポイント一覧

| メソッド | パス | 用途 | 機能 ID |
|----------|------|------|---------|
| POST | `/api/point-applications` | 下書き作成 or 申請送信（`status` 指定） | IN-08 / IN-09 |

### 5.2 検索 API の仕様（補足）

| 画面 | エンドポイント | 検索対象フィールド |
|------|----------------|---------------------|
| S-02 | `GET /api/point-applications?q=...&status=...&tab=...` | `title` / `description` / 申請者氏名 / `ActivityGenre.name` |
| S-04 | `GET /api/approvals?q=...&tab=...&applicant_user_id=...` | 同上 |
| S-06 | `GET /api/notifications?q=...&tab=...` | `title` / `body` / 送信者氏名 |

- `q`: キーワード（部分一致、大文字小文字無視）
- S-04 の `tab` 値: `waiting / completed / all`（§3.7 タブと対応）
- S-06 の `tab` 値: `all / unread`（§3.8 タブと対応）
- S-04 の `applicant_user_id`: 申請者フィルタ。未指定で全員（タブ条件のみ適用）
- ページング: **オフセット方式、`limit=20`（既定）／`offset` で送信**。カーソル方式への切替は将来検討
